# Personal Financial Tracker API

[English version](README.md)

Невеликий RESTful API для персонального трекера бюджету. Проєкт призначений для обліку особистих доходів і витрат, а також для практики базових концепцій бекенд-розробки.

## Ідея

API дозволяє автентифікованому користувачу створювати, переглядати, оновлювати та видаляти фінансові записи. Основна сутність на старті - транзакція: дохід або витрата із сумою, категорією, датою та необов'язковим описом.

## Що потрібно реалізувати

- CRUD для основної сутності: транзакцій.
- Автентифікацію з Better Auth: спочатку email і пароль.
- PostgreSQL як основну базу даних.
- Drizzle ORM для опису схем, міграцій і запитів.
- Zod для валідації вхідних даних.
- Базову обробку помилок і логування.
- Docker Compose для локальної розробки.

## Запропонована модель даних

### Користувачі

- `id`
- `email`
- `passwordHash`
- `createdAt`
- `updatedAt`

### Транзакції

- `id`
- `userId`
- `type` - `income` або `expense`
- `amount`
- `category`
- `description`
- `date`
- `createdAt`
- `updatedAt`

## Приклади ендпоінтів

- `POST /auth/sign-up` - зареєструвати нового користувача.
- `POST /auth/sign-in` - увійти в акаунт.
- `GET /transactions` - отримати список транзакцій поточного користувача.
- `POST /transactions` - створити транзакцію.
- `GET /transactions/:id` - отримати одну транзакцію.
- `PATCH /transactions/:id` - оновити транзакцію.
- `DELETE /transactions/:id` - видалити транзакцію.

## Теми для вивчення в процесі

- Проєктування схеми бази даних: як думати про таблиці, зв'язки, індекси та обмеження.
- HTTP status codes: логіка вибору правильної відповіді, а не просто заучування списку.
- JWT: що це таке, як працює, де зберігати його на клієнті та чому це важливо.
- Змінні середовища та конфігурація: `dotenv`, локальна конфігурація та валідація через Zod.

## Локальна розробка

Запланований локальний запуск через Docker Compose:

```bash
docker compose up
```

Після цього API має бути доступним локально, а PostgreSQL має працювати в контейнері.
