# Personal Financial Tracker API

Невеликий RESTful API для особистого бюджетного трекера. Проєкт призначений для обліку особистих доходів і витрат та практики базових концепцій backend-розробки.

## Ідея

API дозволяє автентифікованому користувачу створювати, читати, оновлювати та видаляти фінансові записи. Основна сутність на старті — транзакція: дохід або витрата, з сумою, категорією, датою та необов’язковим описом.

## Що потрібно реалізувати

- CRUD для основної сутності: транзакції.
- Автентифікація через Better Auth: спочатку email і пароль.
- PostgreSQL як основна база даних.
- Drizzle ORM для опису схем, міграцій і запитів.
- Zod для валідації вхідних даних.
- Базова обробка помилок і логування.
- Docker Compose для локальної розробки.

## Запропонована модель даних

### Користувачі (Users)

- `id`
- `email`
- `passwordHash`
- `createdAt`
- `updatedAt`

### Транзакції (Transactions)

- `id`
- `userId`
- `type` — `income` або `expense`
- `amount`
- `category`
- `description`
- `date`
- `createdAt`
- `updatedAt`

## Приклади ендпоінтів

- `POST /auth/sign-up` — реєстрація нового користувача.
- `POST /auth/sign-in` — вхід у систему.
- `GET /transactions` — список транзакцій поточного користувача.
- `POST /transactions` — створення транзакції.
- `GET /transactions/:id` — отримання однієї транзакції.
- `PATCH /transactions/:id` — оновлення транзакції.
- `DELETE /transactions/:id` — видалення транзакції.

## Теми для вивчення в процесі роботи

- Проєктування схеми БД: як думати про таблиці, зв’язки, індекси та обмеження.
- HTTP-коди статусу: логіка вибору правильної відповіді, а не просто запам’ятовування списку.
- JWT: що це, як працює, де зберігати на клієнті і чому це важливо.
- Змінні середовища та конфігурація: `dotenv`, локальна конфігурація та валідація через Zod.

## Локальна розробка

Запланований локальний запуск через Docker Compose:

```bash
docker compose up
```

Після цього API має бути доступним локально, а PostgreSQL — працювати в контейнері.
