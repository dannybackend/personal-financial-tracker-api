# Backend Developer Roadmap
### React/Next.js → Node.js/TypeScript Backend Engineer
#### Старт: червень 2026 | Горизонт: 7-8 місяців

> **Про документ:** загальний план переходу frontend → backend, читається на старті сесії для контексту. Деталі проєкту — в `AGENTS.md` (правила для агента, SSOT), `DECISIONS.md` (архітектура), `PROGRESS.md` (статус завдань); тут не дублюється.
> **Контекст:** React/Next.js розробник (2-3 роки), в переході на backend. Ціль — продуктова компанія або стартап, роль backend/fullstack інженера. Спілкування українською, технічні терміни англійською.

## Принципи

1. **Проєкти — головна одиниця навчання.** Без туторіалів заради туторіалів — кожна концепція вивчається в контексті реального проєкту.
2. **Розуміння архітектури — вище знання синтаксису.** Навіщо — важливіше ніж як. Синтаксис підкаже агент, архітектурне рішення — твоє.
3. **Глибина там де важливо, швидкість — де можна.** Безпека, дизайн БД, архітектура — глибоко. Інше — достатньо знати що є і вміти поставити правильне питання агенту.
4. **Кожен проєкт — у production-подібному середовищі.** З першого дня: Git flow, Docker, env variables, базовий CI.
5. **План — живий документ.** Фази стискаються/розтягуються/міняють порядок по ходу. Зміни вносяться прямо сюди.

---

## Компроміс: делегувати vs розуміти самому

**Розуміти глибоко (не делегувати):**
- Чому саме така структура БД і які будуть наслідки
- Як компоненти системи взаємодіють між собою (data flow)
- Безпека: де і чому можуть бути вразливості
- Trade-offs: чому це рішення, а не інше
- Читання і розуміння будь-якого згенерованого коду

**Делегувати агенту:**
- Boilerplate, CRUD ендпоінти, middleware шаблони
- Тести по заданій специфікації
- Рефакторинг за архітектурними інструкціями
- Code review (CodeRabbit)
- Документація, JSDoc, генерація типів зі схем

**Правило перевірки:** якщо агент написав 50 рядків — перед тим як рухатись далі, поясни собі (вголос або коментарем) що робить кожен блок і чому. Не можеш — попроси агента пояснити, потім закрий пояснення і напиши сам.

## Технологічний стек

| Шар | Технологія | Чому |
|-----|-----------|------|
| Runtime | Node.js 24 | Знайома екосистема, LTS, нативні API замінюють залежності |
| Мова | TypeScript 5 (strict) | Обов'язково для серйозного бекенду, агенти генерують кращий код з типами |
| Фреймворк | Hono (основний) + Fastify (знати) | Мінімалізм, кожне рішення видно — критично для навчання архітектури |
| ORM | Drizzle ORM | Type-safe, SQL-like, змушує думати в термінах SQL, добре працює з агентами |
| База даних | PostgreSQL | Промисловий стандарт |
| Валідація | Zod | Де-факто стандарт |
| Автентифікація | Better Auth | Сучасна, type-safe заміна Passport.js |
| Кеш / черги | Redis | Кеш + rate limiting + черги |
| Черги завдань | BullMQ | Стандарт для Node.js на Redis |
| Тестування | Vitest | Швидший сучасний аналог Jest |
| Контейнери | Docker + Docker Compose | З першого дня, не як ускладнення |
| CI/CD | GitHub Actions | Простий старт, масштабується |
| Документація API | Scalar + OpenAPI | Сучасна альтернатива Swagger UI |
| Моніторинг | OpenTelemetry + Grafana | Індустріальний стандарт observability |

## Фаза 2 — Реальні системи
**Тижні 9-20**

Мета: те що відрізняє іграшкові API від production систем — черги, кеш, webhooks, складніша архітектура.

**Проєкт 2 — SaaS Boilerplate / Starter.** Найважливіший проєкт плану — портфоліо і можлива основа власного продукту.

Що будувати: multi-tenant архітектура (організації + users) · auth email/password + OAuth через Better Auth · subscription billing через Stripe (webhooks, плани, ліміти) · background jobs через BullMQ (email, обробка файлів) · file uploads (S3-compatible, Cloudflare R2) · admin API, rate limiting per user/organization · structured logging (Pino), OpenTelemetry трейсинг, health checks, graceful shutdown.

Архітектурні рішення — самому, не делегувати:
- Multi-tenancy: row-level security чи окремі схеми?
- Idempotent обробка Stripe webhook подій
- Структура проєкту: по feature чи по layer?
- Де і як кешувати subscription status?

**Checkpoint:**
- [ ] SaaS boilerplate живе в продакшені
- [ ] Stripe checkout реально працює
- [ ] Background jobs виконуються, видимість в логах
- [ ] Архітектурна діаграма яку можеш захистити на інтерв'ю

Пошук роботи паралельно (старт з тижня 22-24, не чекаючи "готовності"): LinkedIn оновлений, Проєкти 1-2 публічні з README, 2-3 технічні пости, ціль — fullstack ролі де цінується React + новий backend досвід.

Наратив на інтерв'ю: *"Frontend розробник який навмисно перейшов на backend щоб розуміти всю систему. Три проєкти з нуля — можу пояснити кожне архітектурне рішення."*

## Як працювати з агентом

Конкретні технічні правила — в `AGENTS.md` проєкту (SSOT), не тут.

**Щодня:**
1. Плануй перед тим як промптити — напиши намір і чому, потім проси реалізувати
2. Один запит — одна відповідальність ("login endpoint що приймає email/password і повертає сесію", не "весь auth модуль")
3. Перечитуй кожен diff перед прийняттям — агент помиляється, особливо з бізнес-логікою
4. Використовуй агента для пояснень: "поясни чому тут потрібна транзакція", "які edge cases я пропустив"

**CodeRabbit (код-рев'ю):**
- Кожна фіча — окрема PR, навіть соло
- Кожен коментар — як від сеньйора: виправляй або аргументовано не погоджуйся, не dismiss мовчки

## Метрики переходу між фазами

**Фаза 1 → 2:** новий Hono/Drizzle проєкт з нуля за 2-3 години без туторіалу; розумієш кожен рядок Проєкту 1; можеш пояснити схему БД і чому саме так.

**Фаза 2 → 3:** можеш намалювати архітектуру Проєкту 2 на дошці і пояснити кожен компонент; дебажив production проблему через логи/трейси; Stripe webhooks надійні і зрозумілі.

**Готовність до пошуку роботи:** 2+ публічні проєкти з README (рішення + чому); можеш вести архітектурне обговорення ("як би ти масштабував це?"); розумієш коли cache/queue/transaction і чому.
