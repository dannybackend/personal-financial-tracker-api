# Навчальний журнал бекенду

> Живий конспект застосованих у проєкті бекенд-концепцій.
> `DECISIONS.md` пояснює важливі архітектурні рішення.
> `LEARNING.md` пояснює, які знання ми вперше застосували, де саме в коді й навіщо.

---

## Як читати

Записи йдуть хронологічно: старі зверху, нові знизу.

Якщо ти вже читав файл раніше, переходь до останнього знайомого запису і рухайся нижче. Ідея документа не в тому, щоб замінити документацію, а в тому, щоб швидко згадати: "що це за концепція, чому вона тут, де я можу подивитись приклад".

## Коли додавати запис

Додаємо короткий запис, коли в коді вперше з'являється нова для навчального проєкту бекенд-концепція:

- новий тип інфраструктури: база, кеш, черга, міграції, Docker
- новий патерн API: route handler, middleware, auth, pagination, validation
- новий тип безпеки: HttpOnly cookie, rate limiting, env validation
- новий підхід до даних: foreign key, index, transaction, enum, UUID, nullable relation
- новий testing/devops-підхід: integration test, CI, OpenAPI, deployment

Не додаємо запис для дрібних правок, перейменувань, форматування або повторного використання вже описаної концепції.

## Формат запису

```md
### Назва концепції

**Де:** `path/to/file.ts`
**Що це:** 1-2 речення простими словами.
**Навіщо в цьому проєкті:** 1-2 речення в контексті фінансового трекера.
**Ключова думка:** одна коротка фраза, яку варто запам'ятати.
```

## Журнал

### Docker Compose для локальної інфраструктури

**Де:** `docker-compose.yml`
**Що це:** Docker Compose описує кілька сервісів для локальної розробки в одному файлі. У нас він піднімає PostgreSQL і Redis без ручного встановлення цих сервісів у систему.
**Навіщо в цьому проєкті:** API одразу має однакове локальне середовище для всіх розробників: база на `localhost:5433`, Redis на `localhost:6379`.
**Ключова думка:** інфраструктура має запускатись командою, а не усною інструкцією.

### Environment variables і валідація конфігурації

**Де:** `.env.example`, `src/db/db.ts`, `src/lib/auth.ts`
**Що це:** Environment variables зберігають конфігурацію, яка відрізняється між середовищами: URL бази, секрети, base URL. Zod перевіряє ці значення на старті, щоб застосунок падав одразу з ясною помилкою, а не ламався пізніше.
**Навіщо в цьому проєкті:** `DATABASE_URL`, `BETTER_AUTH_SECRET` і `BETTER_AUTH_URL` не можна хардкодити в коді, бо локальне, production і test-середовище матимуть різні значення.
**Ключова думка:** конфігурація живе в env, а код має перевіряти, що вона валідна.

### Drizzle schema як опис структури бази

**Де:** `src/db/schema.ts`, `src/db/auth-schema.ts`
**Що це:** Drizzle schema описує таблиці, колонки, типи, constraints, relations та індекси в TypeScript. З цієї схеми генеруються міграції для PostgreSQL.
**Навіщо в цьому проєкті:** фінансові сутності (`users`, `accounts`, `categories`, `transactions`, `budgets`) мають бути описані явно, щоб код і база не жили окремими правдами.
**Ключова думка:** schema-файл це контракт між TypeScript-кодом і PostgreSQL.

### Міграції бази даних

**Де:** `src/db/migrations/0000_smooth_sway.sql`
**Що це:** Міграція це зафіксована зміна структури бази: створити таблицю, додати enum, foreign key або index. Вона дозволяє відтворити однакову базу на іншій машині.
**Навіщо в цьому проєкті:** новий розробник може підняти PostgreSQL і виконати `npm run db:migrate`, щоб отримати потрібні таблиці без ручного SQL.
**Ключова думка:** схема в коді описує бажаний стан, міграції доводять реальну базу до цього стану.

### Foreign key для зв'язків між таблицями

**Де:** `src/db/schema.ts`
**Що це:** Foreign key гарантує, що запис посилається на існуючий запис в іншій таблиці. Наприклад, `transactions.accountId` має вказувати на реальний account.
**Навіщо в цьому проєкті:** транзакція без власника або рахунку не має сенсу, тому база сама захищає нас від "висячих" фінансових записів.
**Ключова думка:** зв'язки між сутностями треба захищати на рівні бази, не тільки в коді.

### Індекси на foreign key колонках

**Де:** `src/db/schema.ts`
**Що це:** Index прискорює пошук і JOIN по колонці. Foreign key часто використовується у запитах типу "всі транзакції цього користувача", тому без індексу база може сканувати всю таблицю.
**Навіщо в цьому проєкті:** фінансовий трекер постійно фільтруватиме дані по `userId`, `accountId`, `categoryId`.
**Ключова думка:** foreign key відповідає за цілісність, index відповідає за швидкість.

### UUID primary keys

**Де:** `src/db/schema.ts`
**Що це:** UUID це непослідовний ідентифікатор, який складно вгадати. На відміну від `serial`, він не показує, скільки записів існує в таблиці.
**Навіщо в цьому проєкті:** id фінансових сутностей не мають розкривати кількість користувачів, рахунків або транзакцій.
**Ключова думка:** UUID зменшує ризик enumeration-атак і краще підходить для публічних API id.

### `numeric(12, 2)` для грошей

**Де:** `src/db/schema.ts`
**Що це:** `numeric` зберігає десяткові числа точно, на відміну від floating point типів. Для грошей це критично, бо копійки не мають губитися через округлення.
**Навіщо в цьому проєкті:** `transactions.amount` і `budgets.amount` описують фінансові суми, тому мають зберігатися без математичних сюрпризів.
**Ключова думка:** гроші не зберігають у `float`.

### Auth schema окремо від доменної schema

**Де:** `src/db/auth-schema.ts`, `src/db/schema.ts`, `src/lib/auth.ts`
**Що це:** Better Auth має свої таблиці для користувачів, сесій, акаунтів провайдерів і verification-токенів. Доменна таблиця `users` зберігає профіль саме фінансового трекера.
**Навіщо в цьому проєкті:** автентифікація і бізнес-дані мають різні відповідальності, тому ми не змішуємо `auth_account` з фінансовими `accounts`.
**Ключова думка:** auth-користувач відповідає на "хто увійшов", domain user відповідає на "чиї це фінансові дані".

### HttpOnly cookie для сесії

**Де:** `src/lib/auth.ts`
**Що це:** HttpOnly cookie зберігає session token так, що JavaScript у браузері не може його прочитати. Це зменшує шкоду від XSS порівняно з токеном у `localStorage`.
**Навіщо в цьому проєкті:** фінансовий трекер працює з чутливими даними, тому сесія має бути захищена базовими web security практиками.
**Ключова думка:** session token краще тримати в HttpOnly cookie, а не в JavaScript-accessible storage.

### Hono server bootstrap

**Де:** `src/index.ts`
**Що це:** Bootstrap сервера створює Hono app, реєструє routes і запускає HTTP-сервер на порту. Зараз є мінімальний `GET /`, який підтверджує, що сервер стартує.
**Навіщо в цьому проєкті:** це точка входу API, до якої поступово будуть підключатися auth routes, CRUD routes і middleware.
**Ключова думка:** `src/index.ts` збирає HTTP-застосунок докупи і запускає його.

### Email enumeration prevention

**Де:** `src/lib/auth.ts`, `src/index.ts`
**Що це:** Enumeration-атака — коли зловмисник дізнається, чи існує конкретний запис (тут — email) в системі, просто спостерігаючи за відмінністю у відповіді, навіть без доступу до пароля чи інших секретів.
**Навіщо в цьому проєкті:** сам факт, що конкретний email зареєстрований у фінансовому трекері, вже чутлива інформація, тому реєстрація не повинна видавати цю різницю у відповіді.
**Ключова думка:** однакова відповідь на "email вже зайнятий" і "успішна реєстрація" — це так само захист, як валідація вхідних даних, тільки навпаки.

### Account linking (один `auth_user`, кілька `auth_account`)

**Де:** `src/db/auth-schema.ts` (таблиця `auth_account`)
**Що це:** Один `auth_user` може мати кілька рядків `auth_account` — по одному на кожен спосіб входу (email/password, Google, GitHub). `account_id` — ідентифікатор на боці провайдера; для email/password дорівнює `user_id`, бо іншого провайдера немає.
**Навіщо в цьому проєкті:** зараз лише email/password, але схема вже готова до OAuth пізніше без міграції — просто додасться ще один рядок `auth_account` з іншим `provider_id`.
**Ключова думка:** автолінкування акаунтів по email — це водночас і зручність, і security-рішення: Better Auth лінкує автоматично, тільки якщо провайдер підтверджує verified email.

### Database hooks (lifecycle-хуки бібліотеки)

**Де:** `src/lib/auth.ts` (`databaseHooks.user.create.after`)
**Що це:** Hook — функція, яку бібліотека сама викликає в певний момент свого життєвого циклу (тут — одразу після створення юзера), без явного виклику з нашого коду.
**Навіщо в цьому проєкті:** синхронізує `auth_user` (Better Auth) і `users` (доменний профіль) автоматично, без окремого API-виклику з клієнта, який можна забути зробити.
**Ключова думка:** hook надійний рівно настільки, наскільки бібліотека гарантує його виконання — варто перевіряти це емпірично, а не вірити на слово документації.

### Dual-write problem (два записи, одна логічна операція)

**Де:** `src/lib/auth.ts` (`databaseHooks.user.create.after` пише в `users` як реакцію на створення `auth_user`)
**Що це:** коли одна логічна дія має записати дані у два різні місця, керовані різним кодом чи системами, — гарантувати, що відбудуться обидва записи або жоден, складно чи неможливо без спеціальних патернів (outbox, reconciliation job, saga).
**Навіщо в цьому проєкті:** пояснює, чому `databaseHooks.user.create.after` не атомарний з самим створенням `auth_user` (докладніше — `docs/DECISIONS.md`). Це не вада конкретно Better Auth: з хостованим провайдером типу Firebase Auth чи Auth0 та сама проблема була б гіршою, бо там це взагалі різні бази даних без жодної спільної транзакції.
**Ключова думка:** неможливість гарантувати два записи атомарно без спільної транзакції — фундаментальне обмеження бекенду, а не недбалість конкретної бібліотеки.
