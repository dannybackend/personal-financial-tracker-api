## Tooling & Agent Setup

### AGENTS.md як SSOT, .cursor/rules — короткі витяжки

**Рішення:** `AGENTS.md` — єдине повне джерело правил (single source of truth) для агентів. `.cursor/rules/*.mdc` — короткі витяжки з `globs`, підвантажуються тільки для релевантних файлів.
**Чому:** уникнути дублювання одних і тих же правил в кількох файлах і ризику розсинхронізації (drift). Економія токенів — Cursor не тягне всі правила в кожен запит.

### Docker Desktop — all-users install

**Рішення:** Docker Desktop встановлений в режимі "install for all users", другий розробник доданий в групу `docker-users`.
**Чому:** контейнери і образи мають бути доступні обом розробникам на одній фізичній машині (Windows, спільний ПК).

### Імпорт AGENTS.md замість ручних витяжок (оновлює запис вище)

**Рішення:** `.cursor/rules/*.mdc` та `.agents/rules/*.md` більше не тримають ручні витяжки з `AGENTS.md`. `core.mdc` і `core.md` тепер містять лише імпорт (`@AGENTS.md` і `@../../AGENTS.md` відповідно), `database.mdc/.md`, `handlers.mdc/.md`, `security.mdc/.md` — видалені. Новий `CLAUDE.md` в корені імпортує `AGENTS.md` через `@AGENTS.md` для Claude Code.
**Чому:** Ревізія показала, що витяжки розійшлися з `AGENTS.md`: інструкції "додай запис в DECISIONS.md" і "познач пункт в PROGRESS.md" є в `AGENTS.md`, але були відсутні в усіх `.mdc`-файлах (лишився тільки Learning Journal). Сесія в Cursor чи Antigravity ніколи не отримувала команду вести DECISIONS.md чи PROGRESS.md. `.agents/rules/` додали пізніше з тим самим дублюванням і тим самим пропуском, без жодного запису тут. Ручна синхронізація кількох копій одних правил ненадійна навіть в одноосібному проєкті — нативний `@file`-імпорт (Claude Code, Cursor, Antigravity) прибирає друге джерело правди фізично, а не дисципліною.
**Компроміс:** `core.mdc`/`core.md` вже були `alwaysApply`/`always_on` — тепер вони підвантажують увесь `AGENTS.md` в кожному запиті, а не тільки умовно релевантні `database.mdc`/`handlers.mdc`/`security.mdc`, як раніше. При поточному розмірі проєкту це прийнятна ціна; якщо `AGENTS.md` суттєво виросте — можна повернути glob-based підвантаження, розбивши файл на кілька імпортованих частин.
**Альтернатива яку відкинули:** залишити `database.mdc`/`handlers.mdc`/`security.mdc`, перетворивши кожен на окремий `@AGENTS.md`-імпорт замість витяжки. Відкинули — `core.mdc` вже підвантажує весь `AGENTS.md` на кожен запит, тому ці файли додавали б той самий вміст ще раз (а для `src/routes/**`, що збігається і з `handlers.mdc`, і з `security.mdc` — потенційно двічі) без жодної нової інформації.

## База даних — загальна структура

### Foreign keys (id), не текстові поля для зв'язків

**Рішення:** `account_id`, `category_id` в `transactions` — integer/uuid посилання, не назви текстом.
**Чому:** зв'язки через id дають цілісність даних (перейменування категорії не ламає історію), швидкий підрахунок (`SUM`/`GROUP BY`), захист від видалення пов'язаних даних через foreign key constraints.

### Баланс рахунку рахується динамічно, не зберігається в `accounts`

**Рішення:** немає поля `balance` в `accounts`. Баланс = `SUM` транзакцій по `account_id`.
**Чому:** уникаємо race conditions при паралельних записах — дві одночасні транзакції можуть перезаписати результат одна одної якщо баланс зберігається окремим полем. Джерело правди — самі транзакції.
**Компроміс:** повільніше при дуже великих обсягах даних — вирішується індексом на `account_id`; за потреби пізніше можна кешувати в Redis.
**Альтернатива яку відкинули:** зберігати `balance` і оновлювати через `SELECT FOR UPDATE` в транзакції — швидше читання, але вищий ризик тихих багів якщо десь забути обгорнути оновлення в транзакцію.

### `category_id` в `transactions` — nullable, `onDelete: set null`

**Рішення:** якщо категорію видалено — транзакція лишається з `category_id = null` ("без категорії"), не видаляється каскадно.
**Чому:** видалення категорії не повинно видаляти історію транзакцій.
**Альтернатива яку відкинули:** системна категорія "Інше" створена автоматично для кожного юзера — бізнес-логічно чистіше, але додає складність (seed-логіка при реєстрації, захист від видалення) не потрібну для MVP.

### `category_id` в `budgets` — required, `onDelete: cascade`

**Рішення:** якщо категорію видалено — пов'язані записи `budgets` видаляються каскадно через `budgets.category_id`.
**Чому:** бюджет без категорії втрачає сенс для планування, на відміну від історичної транзакції.
**Компроміс:** видалення категорії також прибирає її бюджетні ліміти; якщо потрібна історія бюджетів, її треба буде моделювати окремо.

### Cascade delete від `users` до всіх дочірніх таблиць

**Рішення:** видалення юзера каскадно видаляє його `accounts`, `categories`, `transactions`, `budgets`.
**Чому:** дані без власника не мають сенсу існувати в системі.

### Індекси на всіх foreign key колонках

**Рішення:** індекси на `user_id`, `account_id`, `category_id` у всіх таблицях де вони присутні.
**Чому:** без індексу — full table scan при кожному запиті "дані цього юзера" і при JOIN між таблицями. Знайдено CodeRabbit на першому review, виправлено окремою міграцією.

### UUID замість `serial` для всіх primary keys

**Рішення:** всі `id` — `uuid('id').primaryKey().defaultRandom()`, не `serial`.
**Чому:** `serial` розкриває кількість записів (enumeration attack), ускладнює мердж даних з кількох джерел, не можна згенерувати id до запису в базу. Плюс консистентність з Better Auth, який вже використовує текстові/uuid id.
**Коли прийнято:** до появи реальних даних в таблицях — зміна типу primary key після появи даних значно дорожча (треба одночасно конвертувати всі foreign keys).

### `numeric(12, 2)` для всіх грошових полів

**Рішення:** `amount` в `transactions` і `budgets` — `numeric({ precision: 12, scale: 2 })`, не `float`.
**Чому:** `float` має похибку округлення (0.1 + 0.2 ≠ 0.3) — неприпустимо для фінансових розрахунків.

## Автентифікація

### Better Auth з окремими `auth_*` таблицями

**Рішення:** таблиці Better Auth (`auth_user`, `auth_session`, `auth_account`, `auth_verification`) окремо від таблиці `users` (доменний профіль фінтрекера).
**Чому:** уникнути конфлікту назв (`accounts` фінансові рахунки vs `auth_account` OAuth-акаунти в базі Better Auth) і розділити відповідальність — Better Auth керує автентифікацією, `users` — доменними даними.

### Зв'язок `users.auth_user_id → auth_user.id`, one-to-one

**Рішення:** `users` має `authUserId` (unique, FK на `auth_user.id`, cascade delete).
**Чому:** один обліковий запис Better Auth відповідає рівно одному фінансовому профілю.

### Сесія через HttpOnly cookie, не JSON токен в тілі відповіді

**Рішення:** Better Auth сесія через HttpOnly cookie, не токен що зберігається на клієнті в localStorage.
**Чому:** JavaScript не може прочитати HttpOnly cookie навіть при XSS-атаці — безпечніше ніж зберігання токена в localStorage/sessionStorage.

### Створення запису в `users` — через hook, не окремий endpoint

**Рішення:** `onUserCreate` hook в Better Auth автоматично створює запис в `users` одразу при реєстрації. Окремий endpoint (напр. `/api/profile/setup`) — опціонально, для локальної зручності, не є критичним шляхом.
**Чому:** hook атомарний і надійний — неможливо "забути" викликати його з клієнта; уникає розсинхронізації коли є `auth_user` без відповідного профілю в `users`.

### Реєстрація напряму через `/api/auth/sign-up/email`, без кастомної обгортки

**Рішення:** реєстрація йде через змонтований Better Auth handler (`app.on(['POST','GET'], '/api/auth/*', ...)` в `src/index.ts`) і його власний ендпоінт `POST /api/auth/sign-up/email` — без кастомного `/api/auth/register`. `emailAndPassword.autoSignIn` встановлено в `false` (`src/lib/auth.ts`).
**Чому:** дефолтна поведінка Better Auth для вже зареєстрованого email — `422`, що збігається зі статус-кодами проєкту з `AGENTS.md` (`400/401/403/404/422/500` — 409 там немає) без жодної кастомної обгортки. `autoSignIn: false` додатково закриває email-enumeration: без цього sign-up для вже зареєстрованого email повертає відмінну від успіху відповідь (422), дозволяючи зловмиснику перевіряти, які email вже зареєстровані в системі. Перевірено наживо: другий sign-up з тим самим email не створює другий рядок в `auth_user` (Better Auth повертає синтетичний, неперсистований об'єкт з тим самим 200 і формою відповіді).
**Компроміс:** `/api/auth/sign-up/email` стилістично не збігається з плоским REST-стилем решти API (`/accounts`, `/categories`). Після реєстрації користувач більше не отримує сесію автоматично (`token: null` у відповіді) — логін стає обов'язковим окремим кроком, а не опціональною зручністю.
**Альтернатива яку відкинули:** кастомний `POST /api/auth/register` з власними 201/409, що обгортає `auth.api.signUpEmail()` — відкинули, бо довелось би вручну підтримувати ще один шар поверх Better Auth без реальної користі, а 409 не входить у статус-коди проєкту взагалі. Також відкинули `requireEmailVerification: true` як механізм anti-enumeration — теж working, але вимагає реалізованої `sendVerificationEmail` (реальний email-провайдер), якого зараз немає в проєкті; лишається кандидатом на окрему майбутню фічу "email verification".

### Rate limiting: явний `customRules`, in-memory storage

**Рішення:** `rateLimit.enabled: true` + `customRules` для `/sign-up/email` (60с/10) і `/sign-in/email` (60с/5) — суворіше за глобальний дефолт (60с/100). Storage — `memory` (дефолт Better Auth), не Redis.
**Чому:** CodeRabbit review на PR #27 підняв, що `AGENTS.md` вимагає rate limit на auth endpoints, а конфіг його не задавав явно. Better Auth сам вмикає rate limiting за замовчуванням, але лише в production і з одним глобальним лімітом, не заточеним під auth конкретно. Явний `customRules` не додає нової залежності і працює однаково в dev/prod.
**Компроміс:** in-memory storage не координується між репліками — при горизонтальному масштабуванні кожна репліка рахує ліміт окремо, ефективний ліміт множиться на кількість реплік. Прийнятно зараз (один інстанс, Railway/Render), недостатньо, якщо з'являться кілька реплік.
**Альтернатива яку відкинули:** одразу Redis-backed rate limiting зі спільним сховищем між репліками — свідомо не робимо зараз, це окремий тікет #20 (Production Readiness): нова залежність, не MVP-блокер для однорепліковго деплою.

## Відкладені рішення (заплановано на Фазу 2+)

### Merchants — окрема сутність, не в MVP

**Ідея:** таблиця `merchants` (`user_id`, `name`, `icon`) + `merchant_id` в `transactions` — щоб не повторювати вибір іконки для одного й того ж магазину при кожній транзакції.
**Чому відкладено:** для MVP категорій з іконками достатньо. Merchants стане особливо доречним разом з банківською інтеграцією — Monobank/Privat повертають MCC-коди і назви мерчантів автоматично.

### Банківська інтеграція (Monobank / Privat API)

**Ідея:** підключити `accounts` до банківських API для realtime синхронізації транзакцій; ручне управління лишається тільки для готівки.
**Підготовка зараз:** заплановані поля `external_id`, `external_source` в `accounts` — додаються окремою міграцією одразу після базової схеми.
**Чому відкладено:** повна реалізація потребує webhooks + чергу (BullMQ) для idempotent обробки — матеріал Фази 2 roadmap.

### Повторювані транзакції (підписки, оренда)

**Чому відкладено:** потребує логіки розкладу і фонових задач (BullMQ) — Фаза 2 roadmap.
